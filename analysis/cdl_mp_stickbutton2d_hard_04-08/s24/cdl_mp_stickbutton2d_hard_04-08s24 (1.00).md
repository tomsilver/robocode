- v0: only one behavior "press all buttons" (basically ignoring behaviors) -> just tries to motion plan to nearest unpressed button, ignores the table completely so it gets stuck, stick geometry. of note: a ton of magic numbers in `act_helpers.py`, but all properly named and somewhat documented, so that's good
- v1: still only one behavior in `approach.py`, but inside the behavior's step there is a "pick next" loop that actually cycles through (but not a queue): the list of behaviors is: `PressFloorButton` for each button not on table, then after these if buttons on table we add a `GraspStick`, then `PressTableButton` for each table button. So basically it's using the CDL prompting, just not in the way we suggested. Grasp fails: it motion plans to (x = stick right edge, `stick[x] + stick[width]`, y = lower quarter, `stick[y] + stick[height] / 4`). The reason this fails is that it goes right up to the stick, and then spinning and extending the robot arm doesn't work because it gets stuck. Grasp phases:
	- `plan` -> controls other phases and makes motion plan
	- `navigate`-> follows motion plan to grasp point
	- `orient_arm`-> rotates towards stick
	- `activate_vacuum`
  Note that at this phase (and the next) `PressTableButton` is also using a phases approach, where it motion plans to button_x, table_y limit and then tries to rotate the arm towards the button and extend it. this obviously would fail but the agent never gets there to see it.
- v2: correctly identifies grasp is failing, tries to grasp from bottom with arm at pi/2, but fails for the same reason: motion planning too close to the stick and then stuck in rotation. there is a `GRASP_BELOW_OFFSET` for the y now, but I guess not enough. 
- v3: adds a `fine_pos` (fine-tune position) phase to the grasp after navigate: navigate now has loose tolerance for when to stop following the plan (doesn't follow it all the way, when norm < POS_TOL=0.04 moves to fine_pos). The issue is that fine_pos just moves toward the target (bottom center of stick) in a straight line, so if it's a bit off it gets stuck to the side and can't rotate the arm
- v4: retries same strategy but approaching from left (or right as fallback) instead of bottom. still fails because fine_pos gets the agent too close to the stick and it can't rotate properly. Also at this stage `PressTableButton` is rewritten to remove the rotation stuff, now it's just plan path to btn_x, table_y, navigate there, "press" (which tries to extend the arm towards the button).
- v5: this is the one that worked. first of all, the commit strangely says it fixed a `GRASP_SIDE_OFFSET` NameError from v4 (even though it was defined properly in v4) and changed it to  `PRE_GRASP_OFFSET`. what changed is it just got bigger, from 0.107 to 0.120 so now there is enough space to rotate properly and attach the vacuum. This also adds a `push` phase at the end of the grasp, which pushes the robot towards the stick to close the gap after the arm is properly rotated, so it attaches. The new `PressTableButton` without rotation also worked.
- v06 fixes a bug in the PushTableButton: when`PressTableButton.reset()` ran after a previous press, it assumed arm was retracted to ARM_MIN; if the previous press left the arm extended, the stored offset was wrong and the next button would miss. (not sure why it's extending the arm at all since it's grasping from the side, but ok)
- Interesting note about seeds: initially it tested with the usual `[0, 1, 2, 42]`: after v06, everything was solved, so it deliberately expanded to `range(20)` and tested on 20 different seeds, which found some edge cases. It will expand to more later. At this point it was solving 14/20 seeds.
- v07 changes the grasp y pos to be at the bottom instead of bottom + 0.05, so that the stick can reach higher in seeds where the buttons are very far up. Similarly v08 changes some constants to make the robot get as close to the table boundary as possible to reach further up
- v09 adds a ReleaseStick behavior, and PushAllButtons checks if the buttons are reachable with the current grasp side, if not it can drop the stick, regrasp from the other side, and press the remaining ones. Also adds `upward push` phase in PushTableButton to get more stick reach. This is strange since it just extends the robot arm, but doesn't really make sense since it's grasped from the side.
- v10 reverts the upward push since it caused regression
- v11 checks if the arm was extended after `ReleaseStick` and retracts it since otherwise the robot can get stuck
- v12 computes `exclusive` buttons (e.g. ones that can only be reached with grasp from left or grasp from right) and prioritizes getting them first so that only one regrasp is needed
- v13 adds `CenterStick` behavior that repositions stick to the center before `ReleaseStick` and regrasp since otherwise the stick could be too far from one side with not enough space on the other to regrasp it
- v14 changes `FLOOR_MAX_Y`coord to give 0.008 more to try and get more seeds, at this stage it's testing on 5 and 17 (from commit message) -> here it was at 18/20. This change fixed seed 17 but not 5. So it changed to 0.009 more (total of 1.149, was 1.140 before) and it worked for seed 5. It didn't commit this immediately so I found it from the log. this fixed it.
- More seeds: after this, 20/20 were solved, so it started using 20-49 so it had 50 seeds tested in total.
- v15 (final) fixes cases where some floor buttons are in a small pocket between stick and wall (e.g. if stick is very close to one side), in this case the stick needs to be moved first, while the previous approaches motion planned to all floor buttons before grasping, which would've gotten stuck. in these cases, the buttons are marked as "blocked" and are pressed at the end after pushing all table buttons.
- After this, it went on to test from 50-99, found 100/100, then decided to do 100-199, found 200/200, was happy with the approach and ended. final eval is 1.0, used $11.70. I find this behavior of trying more seeds really interesting, perhaps we can make it more principled in the prompt by explicitly asking it to find seeds that contain edge cases. e.g. this final one it found in v15 was pretty tricky.  

Overall, I think the trajectory of bug fixing is pretty cool. The one thing it could have done better was grasp from the bottom instead of the side, most work was to try and reach the buttons that were at the very top, and it basically fixed it by shoving the robot as close to the table as possible. But it worked, so that counts. Otherwise, I'd say that the approach was in a good place by v5, and v6-v15 were just fixing edge case seeds that didn't work immediately.
